# SupportMind

**SupportMind** is a customer-support system that goes beyond answering questions: it **resolves**
tickets end-to-end. It combines a hybrid-retrieval RAG copilot (semantic + BM25 + cross-encoder
reranking, with citations) with a **LangGraph resolution agent** that looks up orders, cancels them,
processes refunds — with a human-approval gate above a configurable dollar amount, durable across a
process restart — and escalates to a human when it can't resolve something safely.

- **Demo video (v2, agent + HITL approvals):** [watch here](https://github.com/Samyak2605/SupportMind/releases/download/demo-v2/Screen.Recording.2026-07-13.at.2.53.41.PM.mov)
- *(an earlier [v1 RAG-only demo video](https://github.com/user-attachments/assets/193999cd-1caa-4cf0-8e4d-04bf320097d1) exists from before the agent layer was added)*
- **Live demo:** not deployed — Render's free tier caps web services at 512MB RAM, and this project's
  retrieval stack (sentence-transformer embeddings + cross-encoder reranker + a 122k-chunk index)
  exceeds that loading into memory (`Ran out of memory (used over 512MB)` per Render's own build log).
  Paying for a bigger instance for a portfolio project didn't seem worth it — the demo video above shows
  the exact same system running against the same code. See [Limitations](#limitations).

## Results at a glance

**Agent behavior (30 scripted scenarios)** — full table + failure analysis: [`results/agent_eval.md`](results/agent_eval.md)

| Metric | Value |
|---|---|
| Task success rate | **97%** (29/30) |
| Tool-call accuracy | 93% |
| Escalation correctness | 97% |
| Grounded-citation rate (kb_question) | 75% |
| Avg reply-quality judge score | 3.40 / 5 |

**Retrieval quality (RAGAS)** — full table: [`results/ragas.md`](results/ragas.md)

| Mode | Reranker | Faithfulness | Answer Relevancy | Context Precision | Context Recall | Rows |
|---|---|---|---|---|---|---|
| semantic | no | 0.85 | 0.70 | 0.90 | 0.84 | 30/30 |
| hybrid | no | 0.86 | 0.70 | 0.93 | 0.85 | 30/30 |
| hybrid | yes | 0.82 | 0.55 | 1.00 | 0.87 | 30/30 |

All 3 variants complete, 30/30 rows, no skipped/NaN cells — reproducible via `scripts/run_evaluation.py`
(a dedicated `GROQ_EVAL_API_KEY` and sequential execution were needed to get a clean run within Groq's
free-tier daily quota; see Design Decisions). One honest observation rather than a victory-lap number:
**adding the reranker raises context precision to a perfect 1.00 and improves context recall, but lowers
faithfulness and answer relevancy slightly** in this sample — reranking optimizes *which* chunks get
retrieved, not what the LLM does with them, and a narrower top-k can occasionally cut a chunk the
generator would have used well. Worth a deeper look before assuming "add a reranker" is a free win.

Both tables are generated by scripts that write to `results/`, never hand-typed — see
[Design Decisions](#design-decisions) for why that matters.

## Architecture

```text
                    User (custom web UI at /, or POST /chat directly)
                                      |
                                      v
                    +--------------------------------------+
                    |         LangGraph resolution agent     |
                    |  understand -> ask | act(<=4) -> guard  |
                    |         -> respond | escalate           |
                    |  brain: Groq llama-3.3-70b-versatile    |
                    |  fallback: Gemini (logged failover)     |
                    +---------------+----------------------+-+
                                    |                       |
                    +---------------v------+   +------------v-----------+
                    | kb_search             |   | order_lookup            |
                    | (wraps the existing   |   | cancel_order            |
                    | hybrid + rerank       |   | refund_initiate*         |
                    | retrieval pipeline)   |   | escalate_to_human        |
                    +---------------+------+   +------------+-----------+
                                    |                        |
                                    v                        v
                    +--------------------------+   +------------------------+
                    | Retrieval layer           |   | Mock order system       |
                    | Chroma + BGE embeddings   |   | SQLite + SQLAlchemy     |
                    | BM25 + RRF + reranker     |   | seeded, 5% failures     |
                    +--------------------------+   +------------------------+

  * refund above config.agent.refund_approval_cap -> interrupt() -> checkpointed via SqliteSaver
    -> GET/POST /approvals -> approve/reject -> graph resumes from checkpoint, even after a
    process restart (proven by tests/test_agent_restart_resume.py). Idempotency key prevents
    double refunds (tests/test_orders_idempotency.py).

  /query (original RAG endpoint) is unchanged. /chat is additive.
```

## Product overview

**Copilot flow (`POST /query`, "Ask" tab):** ask a support question → hybrid retrieval + rerank →
grounded answer with citations, retrieved evidence, and a latency breakdown.

**Resolution flow (`POST /chat`, "Resolve" tab):** describe what you need → the agent classifies intent,
asks for a missing order number if needed, calls the right tool(s), and either resolves it directly,
pauses for human approval (large refunds), or escalates. Try, against the seeded demo data:

- `What's the status of order 5?`
- `Cancel order 12, I changed my mind`
- `Refund order 8, it arrived damaged` (small amount → auto-approved)
- `Refund order 1, it never worked` (large amount → pauses for approval; approve/reject it in the
  "Pending Approvals" panel or via `POST /approvals/{id}`)
- `I want to speak to a human right now` (escalates)

## Tech stack

- **Language:** Python
- **Backend:** FastAPI (`/query`, `/chat`, `/approvals`)
- **Frontend:** a hand-built, dependency-free web UI (vanilla HTML/CSS/JS, no React/Gradio/Streamlit),
  served as static files from the same FastAPI app at `/`
- **Agent orchestration:** LangGraph (`StateGraph`, `interrupt`/`Command` for human-in-the-loop,
  `SqliteSaver` for durable checkpoints)
- **Retrieval:** `BAAI/bge-small-en-v1.5` embeddings + ChromaDB, `rank-bm25`, Reciprocal Rank Fusion,
  `cross-encoder/ms-marco-MiniLM-L-6-v2` reranker
- **LLMs:** Groq `llama-3.3-70b-versatile` (agent brain), Groq `llama-3.1-8b-instant` (existing `/query`
  generation, unchanged), Gemini (`gemini-flash-lite-latest`, agent fallback + eval simulated customer)
- **Mock order system:** SQLAlchemy models on SQLite, seeded, with injected failure/latency
- **Evaluation:** RAGAS (retrieval) + a custom scripted-scenario harness (agent behavior)
- **CI/CD:** GitHub Actions, Docker, Render

## Local run guide

```bash
python -m venv .venv
source .venv/bin/activate
PYTHONPATH=src .venv/bin/pip install --no-cache-dir -r requirements.txt

cp .env.example .env
# add GROQ_API_KEY (required) and GEMINI_API_KEY (required for the agent fallback + eval harness)

PYTHONPATH=src .venv/bin/python scripts/build_index.py   # builds the retrieval index (~10 min, one-time)
PYTHONPATH=src .venv/bin/python scripts/run_api.py        # serves API + web UI on http://localhost:8000
```

Open `http://localhost:8000` for the UI (Ask / Resolve tabs). The orders DB seeds itself automatically on
first startup (50 customers / 200 orders, idempotent — see `AgentRuntime`). `scripts/seed_orders.py` still
exists as a standalone utility if you want to seed without starting the server.

```bash
# tests (fast subset used in CI, no model downloads needed)
PYTHONPATH=src .venv/bin/python -m pytest tests/ -m "not requires_index" -q

# full test suite (needs the built index)
PYTHONPATH=src .venv/bin/python -m pytest tests/ -q

# publish retrieval eval results to results/ragas.md
PYTHONPATH=src .venv/bin/python scripts/run_evaluation.py

# publish the 30-scenario agent eval to results/agent_eval.md (needs GEMINI_API_KEY)
PYTHONPATH=src:. .venv/bin/python scripts/run_agent_evals.py
```

## Deploying (Render)

1. Push this repo to GitHub (make sure `data/raw/bitext_support.csv` is committed — Render builds from
   git, not your local filesystem, and the Docker build needs it to bake the retrieval index).
2. In Render: **New → Blueprint**, point it at this repo (`render.yaml` is already configured for a
   Docker web service on the free plan with a `/health` check).
3. Set the `GROQ_API_KEY` and `GEMINI_API_KEY` environment variables in the Render dashboard (marked
   `sync: false` in `render.yaml` so Render prompts for them rather than committing secrets).

**Note:** the build succeeds, but the service currently fails to *start* on Render's free plan —
it exceeds the 512MB RAM cap loading the embedding + reranker models and the index. See
[Limitations](#limitations) for the measured cause and what a real fix would involve (bigger instance,
smaller/quantized models, or lazy-loading). This repo's live demo is the recorded walkthrough instead.

## Design Decisions

**A hand-built frontend instead of Gradio/Streamlit.** The API layer (`/query`, `/chat`, `/approvals`) is
fully decoupled from the UI, so the frontend is "just" a static HTML/CSS/JS app making `fetch()` calls —
no build step, no Node toolchain, no extra runtime dependency, and full control over the visual design
instead of inheriting a demo-tool look. It's mounted as static files on the same FastAPI app (`app.mount("/",
StaticFiles(...))`), so there's exactly one process and one port in both local dev and production.

**LangGraph over a hand-rolled agent loop.** The graph makes routing explicit and testable: each node
(`understand`, `ask`, `act`, `guard`, `respond`, `escalate`) is a plain function taking `(state, deps)`,
independently unit-testable without touching the LLM. The alternative — a single ReAct-style tool-calling
loop — is more flexible but harder to reason about for an interview walkthrough and harder to guarantee
"missing order id always asks, never guesses."

**SQLite for orders, checkpoints, and approvals — no external database.** This is a portfolio deployment
target (Render free tier, zero external infra), and SQLite's durability is exactly what the
restart-resume test needs to prove: close one `SqliteSaver` connection, open a brand-new one against the
same file, and resume. A production deployment would swap this for Postgres; the `OrdersClient`/checkpointer
boundary is already where that swap would happen.

**Human-approval gate on large refunds, not just a policy in a prompt.** An LLM prompted "ask for approval
above $200" can be talked out of it. `refund_initiate` enforces the cap in code and calls
`langgraph.interrupt()` — the graph literally cannot proceed past that point without a `POST
/approvals/{id}` call recording a human decision. This is the difference between a suggestion and a gate.

**The knowledge base corpus is the public Bitext customer-support dataset**, not proprietary data — stated
plainly rather than obscured. The differentiator in this project is the agent, HITL approval mechanics,
and the dual eval harness, not the dataset.

**Three distinct model configurations in the eval harness** (agent brain: Groq 70b; simulated customer:
Gemini; judge: Groq 8b) so no model grades its own homework, and **task success is a direct SQLite
assertion, never an LLM's opinion** — see `evals/runner.py::evaluate_result`. The judge only scores softer
qualities (reply tone/clarity) that genuinely need semantic judgment.

**RAGAS judge model deliberately different from the agent's own brain model, plus a dedicated eval API
key.** The 70b model used for the agent's brain has a much smaller free-tier daily token quota than the
8b model, and a full RAGAS pass (30 rows × 3 retrieval variants × 4 metrics) is easily large enough to
exhaust it — which would also starve the live agent demo of its own brain model for the rest of the day.
The fix is two-layered: `GROQ_EVAL_API_KEY` (`EnvironmentSettings.for_eval()`) isolates all eval-batch
traffic onto its own key, separate from the demo key's quota, and the RAGAS judge itself runs on
`llama-3.1-8b-instant` rather than the 70b brain model, since judging doesn't need frontier-model
capacity. See Limitations for the quota numbers this is designed around.

## Failure Analysis

Of 30 scripted scenarios, 29 passed their deterministic task-success check (a direct SQLite assertion,
not an LLM's opinion). The one failure, from [`results/agent_eval.md`](results/agent_eval.md):

- **s27** (kb_question) — goal: *"Ask how to reach customer support directly."* The agent escalated to a
  human instead of answering directly ("I've escalated this to a human specialist..."). The judge's note:
  *"vague and doesn't provide direct contact information."* Root cause: the knowledge base's contact-info
  entries are template-heavy (`{{Customer Support Phone Number}}`-style placeholders — see Design
  Decisions on the Bitext dataset), and the agent's `kb_search` path correctly recognized it had no
  concrete, customer-ready answer and escalated rather than fabricate one. Arguably the *safe* failure
  mode, but not what the customer asked for — a real, disclosed edge case rather than a hidden one.

## Limitations

- **The knowledge base is a public benchmark dataset (Bitext), not real support tickets** — see Design
  Decisions above.
- **Render's free tier has an ephemeral filesystem.** `data/orders.db`, `data/checkpoints.db`, and the
  LLM dev cache are lost on every restart/redeploy — fine for a demo (the DB reseeds itself), but any
  approval left pending across a restart is lost, not just paused. A production deployment needs a
  persistent disk or an external Postgres for the orders DB and checkpointer.
- **Groq's free tier enforces per-model daily token quotas** — 100k tokens/day for the 70b model, 500k
  for the 8b model — that a full RAGAS batch (30 rows × 3 variants × 4 metrics) can approach. A dedicated
  eval API key and running the judge on the 8b model (see Design Decisions) keep this from affecting the
  live agent demo, which uses far less volume per request. A sustained high-traffic demo could still hit
  the same wall on a free-tier key.
- **Not deployed to Render**: the free tier's 512MB RAM cap is exceeded by the retrieval stack
  (embeddings + reranker + torch + a 122k-chunk index) at container startup — confirmed by Render's own
  build log (`Ran out of memory (used over 512MB)`), not a hypothetical. Fixing this for real would mean
  a smaller/quantized embedding model, lazy-loading the reranker, or simply paying for more RAM — a
  reasonable next step for production, not justified for a portfolio demo. The demo video shows the
  identical system running locally against the same code and config.
- **Cold Docker builds are slow** (the retrieval index is baked at build time, ~10 minutes) — true
  regardless of where it's hosted, noted honestly rather than hidden.
- **No authentication** on `/chat`, `/approvals`, or the web UI — anyone with the URL can approve
  refunds. Acceptable for a portfolio demo; not for production.
- **The reranker/retrieval stack was extended rather than rewritten** when the agent layer was added, so
  its existing dedup/intent-filtering behavior (undocumented quirks and all) carries forward unchanged.

## Testing

- `tests/test_agent_restart_resume.py` — **mandatory**: an approval interrupt survives closing one
  `SqliteSaver` connection and opening a new one against the same file, then resumes correctly.
- `tests/test_orders_idempotency.py` — **mandatory**: calling the refund tool twice with the same
  idempotency key produces exactly one `Refund` row, completed exactly once.
- `tests/test_agent_tools.py`, `tests/test_agent_graph.py` — tool-level and graph-level coverage
  (missing-order-id ask flow, cancel eligible/ineligible, auto-refund, escalation after repeated
  transient OMS failures, kb_question grounding), using a `FakeLLMClient` so CI needs no API keys.
- `tests/test_hybrid.py` — original RRF ordering test, untouched.

CI (`.github/workflows/ci.yml`) runs ruff + the deterministic subset (`-m "not requires_index"`) on every
push, with no secrets required. Tests marked `requires_index` and the live eval scripts run locally.

## Recruiter Notes

This project demonstrates:

- Extending a working RAG system into a **tool-executing, stateful agent** (LangGraph) without touching
  its existing, tested retrieval/generation contract
- **Human-in-the-loop safety mechanics** that are actually enforced in code (interrupt + durable
  checkpoint + idempotency), not just described in a prompt
- A **dual evaluation harness** — RAGAS for retrieval, a from-scratch scripted-scenario suite with a
  simulated customer and an independent judge for agent behavior — with task success always grounded in
  real database state
- Designing around **real infrastructure constraints** (provider rate limits, free-tier memory caps)
  rather than only working in a mocked, idealized environment
- CI, Docker, and a documented, honest set of limitations
